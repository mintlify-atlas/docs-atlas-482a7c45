## Overview

The User Management Microservice uses JWT (JSON Web Tokens) for stateless authentication. This guide explains how authentication works, from user signup to token generation and validation.

## Authentication Flow

<Steps>
  <Step title="User Sign Up">
    A new user registers with email, password, and optional phone numbers. The system generates a JWT token upon successful registration.
  </Step>

  <Step title="Token Generation">
    The system creates a signed JWT token containing the user's email (username) as the subject, with an expiration time.
  </Step>

  <Step title="Token Usage">
    The client includes the JWT token in subsequent requests to authenticate the user.
  </Step>

  <Step title="Token Validation">
    The system validates the token signature and expiration before processing authenticated requests.
  </Step>
</Steps>

## JWT Service Implementation

The JWT functionality is implemented through the `JwtService` interface and `DomainJwtService` implementation.

### JwtService Interface

```java JwtService.java
package com.fbaron.core.service.jwt;

public interface JwtService {

    /**
     * Creates a new Jwt with the provided Jwt details.
     *
     * @param username The username of the user to create the Jwt.
     * @return The created String with the Jwt details.
     */
    String generateToken(String username);

    /**
     * Extracts the username from the provided Jwt.
     * @param token The Jwt to extract the username from.
     * @return The extracted username.
     */
    String extractUsername(String token);

    /**
     * Validates the provided token.
     * @param token The token to validate.
     * @return boolean value indicating if the token is valid or not.
     */
    boolean validateToken(String token);

}
```

### Token Generation

The `DomainJwtService` class implements token generation using the JJWT library:

```java DomainJwtService.java
package com.fbaron.core.domain.jwt;

import com.fbaron.core.service.jwt.JwtService;
import com.fbaron.data.config.jwt.JwtConfig;
import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.security.Keys;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.time.Instant;
import java.util.Base64;
import java.util.Date;

@Slf4j
@Service
@RequiredArgsConstructor
public class DomainJwtService implements JwtService {

    private final JwtConfig jwtConfig;

    @Override
    public String generateToken(String username) {
        Instant now = Instant.now();
        Instant expiryDate = Instant.now().plusSeconds(jwtConfig.getExpiration());
        var secretKey = Keys.hmacShaKeyFor(Base64.getEncoder().encode(jwtConfig.getSecret().getBytes()));
        return Jwts.builder()
                .issuer(jwtConfig.getIssuer())
                .subject(username)
                .issuedAt(Date.from(now))
                .expiration(Date.from(expiryDate))
                .signWith(secretKey)
                .compact();
    }

    @Override
    public String extractUsername(String token) {
        return getClaims(token).getSubject();
    }

    @Override
    public boolean validateToken(String token) {
        try {
            getClaims(token);
        } catch (Exception e) {
            log.error("Error validating token: {}", e.getMessage());
            return false;
        }
        return true;
    }

    private Claims getClaims(String token) {
        var secretKey = Keys.hmacShaKeyFor(Base64.getEncoder().encode(jwtConfig.getSecret().getBytes()));
        return Jwts.parser()
                .verifyWith(secretKey)
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }
    
}
```

<Note>
  The token expiration is calculated in seconds, not milliseconds. The configuration value in YAML is in milliseconds but converted to seconds when used.
</Note>

## JWT Token Structure

A generated JWT token contains the following claims:

<ParamField path="iss" type="string">
  **Issuer**: Identifies the application that issued the token.
  
  Example: `fbaron.com`
</ParamField>

<ParamField path="sub" type="string">
  **Subject**: The username (email) of the authenticated user.
  
  Example: `user@example.com`
</ParamField>

<ParamField path="iat" type="number">
  **Issued At**: Timestamp when the token was created.
  
  Example: `1700000000`
</ParamField>

<ParamField path="exp" type="number">
  **Expiration**: Timestamp when the token expires.
  
  Example: `1700086400`
</ParamField>

### Example Token

Here's what a decoded JWT token looks like:

```json
{
  "iss": "fbaron.com",
  "sub": "john.doe@example.com",
  "iat": 1700000000,
  "exp": 1700086400
}
```

## REST Adapters

The authentication endpoints are implemented in the web layer:

### SignUp REST Adapter

```java SignUpRestAdapter.java
package com.fbaron.web.rest;

import com.fbaron.core.service.user.UserService;
import com.fbaron.web.dto.user.SignUpDTO;
import com.fbaron.web.dto.user.UserDTO;
import com.fbaron.web.mapper.UserDtoMapper;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

import javax.validation.Valid;

@Slf4j
@RestController
@RequestMapping("/api/v1/sign-up")
@RequiredArgsConstructor
public class SignUpRestAdapter {

    private final UserService userService;
    private final UserDtoMapper userDtoMapper;

    @PostMapping
    public ResponseEntity<UserDTO> signUp(@RequestBody @Valid SignUpDTO dto) {
        log.debug("Received a request for POST /api/v1/sign-up EndPoint with RequestBody: {}", dto);
        var user = userDtoMapper.dtoToModel(dto);
        user = userService.signUp(user);
        var userDTO = userDtoMapper.modelToDTO(user);
        log.debug("Returning response for POST /api/v1/users EndPoint: {}", userDTO);
        return ResponseEntity.status(HttpStatus.CREATED).body(userDTO);
    }

}
```

### Login REST Adapter

```java LoginRestAdapter.java
package com.fbaron.web.rest;

import com.fbaron.core.service.user.UserService;
import com.fbaron.web.dto.user.LoginDTO;
import com.fbaron.web.dto.user.UserDTO;
import com.fbaron.web.mapper.UserDtoMapper;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

import javax.validation.Valid;

@Slf4j
@RestController
@RequestMapping("/api/v1/login")
@RequiredArgsConstructor
public class LoginRestAdapter {

    private final UserService userService;
    private final UserDtoMapper userDtoMapper;

    @PostMapping
    public ResponseEntity<UserDTO> login(@RequestBody @Valid LoginDTO dto) {
        log.debug("Received a request for POST /api/v1/login EndPoint with RequestBody: {}", dto);
        var user = userService.login(dto.getToken());
        var userDTO = userDtoMapper.modelToDTO(user);
        log.debug("Returning response for POST /api/v1/login EndPoint: {}", userDTO);
        return ResponseEntity.status(HttpStatus.OK).body(userDTO);
    }

}
```

## Token Validation

The `validateToken` method verifies:

1. **Signature Verification**: Ensures the token was signed with the correct secret
2. **Expiration Check**: Validates the token hasn't expired
3. **Claims Parsing**: Verifies the token structure is valid

```java
@Override
public boolean validateToken(String token) {
    try {
        getClaims(token);
    } catch (Exception e) {
        log.error("Error validating token: {}", e.getMessage());
        return false;
    }
    return true;
}
```

<Warning>
  Invalid tokens return `false` instead of throwing exceptions, allowing graceful error handling.
</Warning>

## Security Best Practices

<CardGroup cols={2}>
  <Card title="Secure Secret" icon="key">

## Error Handling

The authentication endpoints return appropriate HTTP status codes:

| Status Code | Description |
|-------------|-------------|
| `201 Created` | User successfully registered |
| `200 OK` | User successfully logged in |
| `400 Bad Request` | Invalid request payload or validation errors |
| `401 Unauthorized` | Invalid token or authentication failed |
| `403 Forbidden` | Token expired or insufficient permissions |
| `404 Not Found` | User not found |
| `409 Conflict` | User with email already exists |
| `500 Internal Server Error` | Server-side error |
